## [0.16.12] - 2026-07-21

### Security
- MCP 单条 JSON 请求上限从 256 MiB 下调到 32 MiB：整条请求作为一行读入内存，故该上限即「单连接
  请求缓冲峰值」。32 MiB 对 `write_file` 的文本/源码同步已足够（真正的大文件/二进制走
  `copy_to_remote` 的分块字节流，不塞进 JSON 行），又把「最多 32 条连接同时灌满大请求」的最坏
  并发内存从 ~8 GiB 压到 ~1 GiB。该 socket 可能经 SSH 反向转发暴露到远端主机，攻击面不止本机用户

### Fixed
- 文件列表的修改时间改为**逐时间戳**按当时真实的时区/夏令时换算：原先取「当前」UTC 偏移缓存起来
  套到所有文件，跨夏令时边界的历史文件会差 1 小时。现 unix 用 `localtime_r(secs)`、Windows 用
  `SystemTimeToTzSpecificLocalTime`，各自按该时刻真实的 DST 状态换算
- 设置保存失败（语言 / 下载目录 / MCP 同意态 / 主题 / 缩放等，因磁盘满 / 只读 / 权限没落盘）现在
  会弹顶部 toast 提示，不再是「UI 显示已切换、实际没保存、重启又变回旧值」的静默失败（尤其 MCP
  同意态这类安全相关设置，静默失败会造成开/关的认知偏差）

## [0.16.11] - 2026-07-21

### Security
- **同时开着多个 iShell 时，AI 可能在一次对话中途静默换一台机器执行命令**：代理进程此前按
  socket 文件的 **mtime** 挑实例（"谁后启动挑谁"），而且**每次工具调用都重挑一遍**。同时每个
  iShell 实例的会话 uid 都从 1 开始独立分配（`next_uid: 0` 起步），于是：AI 拿着实例 A 的
  `uid=1` 干活，用户中途新开了一个 iShell B，下一次调用就跟着跑到 B 上去了——而 B 里 `uid=1`
  同样存在，只是另一台完全不相干的主机。命令照跑，**不报任何错**。只读类操作（`read_screen`/
  `read_file`/`read_history`）不受授权门禁约束，会安安静静读错主机。
  现在：一个 `ishell-mcp` 代理进程**一辈子只绑定一个 iShell 实例**，绑定在首次连接时确定，
  之后永不静默改绑；每条请求都点名目标实例（`McpRequest.instance`），**由被叫的实例自己校验**
  （`McpRequest::is_addressed_to`）——校验放在权威侧而不是代理侧，代理的路径缓存无论怎么失效
  或指错，请求都到不了错误的实例身上。「没点名」一律拒绝，不当成「随便谁都行」。
  绑定的实例找不到了（比如它退出了）会明确报错，而不是顺手改绑到别的实例上。
- **AI 可以不经任何确认，直接往用户自己打开的会话里执行命令**（用户和 AI 同时能往同一个
  shell 里打字）：`open_session` 那道「让用户当面确认」的门禁只挂在开会话这一条路上，而
  `run_command`/`send_input`/`interrupt`/`write_file`/`copy_to_remote` 等全都是拿 `session_uid`
  直接找会话就干活——`ai_owned` 这个标记此前**只有 `close_session` 一处在检查**。于是 AI 只要
  `list_sessions` 看到用户已经打开的会话，直接 `run_command` 上去即可，完全绕过确认。雪上加霜
  的是 `McpSessionInfo` 根本没有 `ai_owned` 字段，AI 连"这是用户的会话、不该碰"都无从判断。
  而用户的会话没有 `ai_owned` 会话那层「不转发用户键盘输入」的只读保护，两路输入会在同一个
  shell 里交织：轻则互相打断，重则把 `run_command` 用来判断命令结束的哨兵标记行搅乱。
  现在：写入类操作若目标是用户自己打开的会话，弹窗让用户当面授权一次（授权按会话 uid 记住，
  内存态，重启 iShell 失效），弹窗里会写明 AI 具体要执行的命令/要覆盖的路径，而不是只说一句
  「AI 想操作这个会话」；`copy_between_sessions` 的源和目标两个会话都要授权（直连模式会往源
  主机落临时私钥、往目标主机的 `authorized_keys` 临时写一行，两边都在改远端状态）。只读操作
  （`read_screen`/`read_history`/`read_file`/`poll_run`）不受限——它们不碰用户的 shell、也不改
  远端状态。`close_session` 维持原样的**直接拒绝**（关闭权限不应该超过打开权限，不接受授权）

### Added
- **多开 iShell 时，用鼠标点窗口来决定 AI 操作哪一个**：代理发现本机有多个 iShell 实例时，
  会向每一个都发一条绑定请求，于是**每个 iShell 窗口上都弹出确认框**，用户在想用的那个窗口上
  点「允许」即可；代理拿到第一个应允就挂断其余连接，落选窗口的框自动消失，不用挨个去点拒绝。
  只有一个实例时直接绑定、全程无感，和以前一样。选择因此发生在用户的眼睛和鼠标上，而不是配置
  文件里——实例标识是纯内部的（`<pid>-<8 位随机>`），不出现在任何 UI 上、也不可配置：用户本来
  就是看着窗口决定的，不该逼他先给窗口取个名字再把名字念给 AI 听。`ISHELL_MCP_SOCKET` 保留
  作为脚本化场景的逃生口，指定了就不弹窗。
- `list_sessions` 返回值新增 `ai_owned` 字段，AI 终于能分辨哪些会话是用户自己打开的；工具
  描述与服务器说明也相应改口：此前写的是「list_sessions 看有哪些已打开的会话 → 需要的目标
  **不在列表里时**再 open_session」，等于明说「列表里有就直接拿来用」，现在改为默认给自己开
  专用会话、不要接管用户的窗口

### Removed
- **回退 0.16.10 引入的 vendor `egui-winit` 补丁**（连同 `vendor/egui-winit` 整个目录）：实测
  证明它在目标环境下**从未生效**，0.16.10 记载的根因是错的。真实根因不在 iShell 也不在
  egui-winit，而在输入法侧——fcitx4 的 XIM 模块默认 `UseOnTheSpotStyle=False`，此时它只向应用
  宣告 `PreeditPosition`/`PreeditNothing` 这类 style（语义是「组字文本由输入法自己画，不交给
  应用」）。winit 因此退回 root-window 模式，iShell 全程收不到**任何** `Ime::Preedit` 事件，
  那个补丁所在的分支自然永远执行不到。诊断方式：直接 `XGetIMValues(XNQueryInputStyle)` 把
  输入法**实际宣告**的 style 位图打出来，一次就问清了此前三轮推断都没问对的问题
- 用户侧解法与其代价（不涉及代码改动）：把 fcitx 的 `UseOnTheSpotStyle` 改为 `True` 并**完整
  重启** fcitx（该选项只在 XIM 服务器启动时读一次），Shift 丢字回显即消失；但随之而来的是
  候选窗口被钉死在窗口左下角、无法跟随光标——这是 libX11 在启用 preedit callbacks 时
  `XNSpotLocation` 失效导致的，见 freedesktop bug #1580（2004 年至今未修），winit 亦在
  `ime/context.rs::set_spot` 的注释里明确记载了这一限制。X11 + XIM 下二者不可兼得，两边都
  不在本项目可控范围内；Wayland 的 `text-input-v3` 没有这个包袱（eframe 已编入 wayland 特性）

### Fixed
- 滚动鼠标滚轮的过程中顺手按下 Ctrl，终端会莫名放大/缩小（几率不高）：Ctrl+滚轮缩放此前
  判断的是 `i.modifiers`（**本帧末**的全局修饰键状态）配 `smooth_scroll_delta`（**带惯性**，
  一次真实滚动之后会连续多帧持续吐出衰减的非零值）。于是「滚完轮子、惯性还在吐值的那几帧
  里按下 Ctrl」就会凭空触发缩放——要恰好按在惯性窗口内，所以几率不高。现改为只看本帧真实
  滚轮事件、并用事件**自带的**修饰键状态（`MouseWheel.modifiers` 记录的是该事件发生那一刻
  的按键状态）；顺带也修掉了「一次 Ctrl+滚动被惯性放大成连续多帧、每帧各缩放一级」的问题
- **AI/MCP 反向转发的 socket 在长连接下会被误删**：崩溃兜底清扫按 mtime 删「超过 24h 的残留
  socket」，但 Unix socket 文件的 mtime 是创建时刻、**不随流量刷新**——于是一条持续存活超过
  24h 的连接，其 socket 会被后来任一条新连接的清扫误删，导致该 iShell 实例在 MCP 里静默失联。
  现由活连接每 6h `touch` 自己的 socket 当心跳，mtime 变成「最后存活时间」：活的永远刷新、
  永不被误删；崩溃的停止刷新、超 24h 自然回收。单次 touch 失败（如通道打开撞服务端 MaxSessions）
  容错继续，不再一次瞬时失败就永久停掉心跳
- **文件列表刷新后偶发「外部新建的同名文件夹不显示、却能在过滤框搜到」**：同一目录可能有多个
  列目录请求在飞（删除后自动刷新 + 手动刷新 + 弱网超时重发），各自独立异步返回、可能乱序，而
  结果此前「谁最后到谁覆盖」——陈旧结果后到就盖掉了较新的列表。现给每个列目录请求打全局单调
  序号，只接受序号不更旧的结果、丢弃乱序陈旧结果；并顺带修掉「已有数据的目录其 loading 标记
  可能永久卡住、压制后续预取」的问题
- **文件夹压缩下载覆盖已存在目标时，中途失败可能连旧目标一起丢失**：此前是「先删旧目标、再把
  解压好的新目录 rename 上去」，一旦删成功、rename 失败就两头皆空、旧数据不可恢复。现改为「旧
  目标先 rename 到同目录备份名（原子、可回滚）→ 换上新目录 → 成功才删备份、失败则把备份换回」，
  保证成功前不破坏原目标
- **导入 `~/.ssh/config` 里的 IPv6 ProxyJump 会生成错误配置**：跳板机地址此前用 `split_once(':')`
  在第一个冒号处拆 host/port，`[2001:db8::1]:2222` 被拆成 host=`[2001`、端口解析失败静默回退 22，
  存成一条连不上或连错端口的跳板机。现正确处理方括号形式 `[v6]:port` 与裸 IPv6 字面量
- 端口转发监听地址对 IPv6 字面量改用结构化构造（`[::1]:8080` 而非手工拼接的 `::1:8080`），避免
  UI 显示与严格解析处因缺方括号出错
- 一批设置（语言/下载目录/强制 X11/MCP 同意态/主题/缩放等）保存失败此前被 `let _ =` 静默吞掉——
  磁盘满/只读/权限导致没落盘时，UI 显示已切换、重启却变回旧值且毫无线索。现失败会 `log::warn!`
  记录（尤其 MCP 同意态这类安全相关设置至少可查证），UI 层弹提示留作后续

### Added
- 终端新增按键/文本/IME 事件的诊断日志（`RUST_LOG=ishell=trace`，只记这三类不刷屏）：
  输入法类问题光看现象只能靠猜，必须先知道输入法到底发了什么、走的哪条通道
- **MCP 协议版本握手**：GUI 与 `ishell-mcp` 代理的线协议是配套编译的、必须同版本。代理现在在
  连接时校验对端 iShell 的协议版本（`MCP_PROTOCOL_VERSION`），不一致就明确报「请重新部署
  ishell-mcp」，而不是静默出错——根治「升级了 iShell 却忘了同步更新代理」这类隐患。
  `ishell-mcp --version` 会打印其 crate 与协议版本
- **代理二进制的标准安装位置 `~/.ishell-mcp/bin/ishell-mcp`** 与一键脚本 `scripts/install-mcp.sh`
  （临时名 + 原子 rename，避免覆盖运行中的二进制报 "Text file busy"，并打印 `claude mcp add`
  注册命令）。README（中/英）与 BUILD.md 更新为新装法，BUILD.md 补充了用 zig/cargo-zigbuild
  把代理交叉编译到其它 Linux 架构的说明（macOS 因需 Apple SDK 仍走原生/CI 构建）

## [0.16.10] - 2026-07-16

### Fixed
- **中文输入过程中按下 Shift，已输入的拼音会从输入框里消失**（再敲/删一个字符才重新冒出来），
  软件内所有输入框（终端、编辑器、路径栏、各对话框）都受影响。根因在上游 `egui-winit`：
  它把「带文本但没带光标位置」的组字事件（`Ime::Preedit(text, None)`）里的 text 整个丢掉，
  并当作组字结束推送 `ImeEvent::Disabled`；而 X11/fcitx 在组字**进行中**按下 Shift 等修饰键
  时发的正是这种事件——组字并没结束，只是这一拍不报光标位置。应用侧收到 Disabled 就把待
  组字文本从输入框移除，于是拼音「消失」；下一次按键带来带光标的 Preedit（内容是完整串），
  之前的字符又一起冒出来。现 vendor 一份 egui-winit 打补丁（同 glutin/vt100 的既有做法，
  见 `vendor/egui-winit/ISHELL_PATCH.md`）：组字文本非空即照常转发 Preedit，仅空文本才当
  组字结束；upstream 针对 macOS backspace 的分支不受影响
- 同时连着多个会话时，打开的文件可能被填进**另一个会话**的编辑器标签里（内容/路径来自
  错误的服务器）：打开文件的请求 id 取自各会话**独立**的 `next_xfer` 计数器，跨会话必然
  重号，而编辑器标签是全局一张表，回填时却只按 id 找标签、没校验会话。两个文件同名同内容
  时表面完全看不出来，但保存会写到**错误的服务器/路径**。现在内容就位、下载进度、加载失败、
  PDF 页数、docx 字节与其后台解析结果等各条回填路径统一按 `(uid, id)` 匹配

## [0.16.9] - 2026-07-16

### Fixed
- **编辑器粘贴含中文的内容会整个应用闪退**（未保存的改动一并丢失）：编辑器同一帧里的顺序是
  「按内容版本重算 lint 缓存 → handle_input 处理粘贴改内容 → 绘制」，于是本帧绘制用的仍是
  改动**之前**算出的 `lint_ranges`——那是一组旧内容的字节偏移。纯 ASCII 时旧偏移顶多让下划线
  画错一帧（每个字节都是字符边界），内容含中文等多字节字符时，旧偏移会落进某个字符中间，
  高亮器按该边界切片直接 panic（`end byte index N is not a char boundary`）。现把这些按内容
  版本缓存的重算（lint / 跨行高亮状态 / 行缩进）移到 `handle_input` 之后、绘制之前，缓存与
  被绘制的内容永远同一版本；同时给高亮器自身加了字符边界吸附作为防御网——UI 绘制路径上的
  panic 会连带丢掉用户未保存的改动，代价远高于「下划线范围略有出入」
- 从终端复制被「软换行」折断的长行时，粘贴出来凭空多了换行符，把一条命令/URL 拆断：
  选区取文本时无条件在屏幕行之间补 `\n`，没区分「终端因行满自动折行」与「真实收到 \n」。
  现按 vt100 的每行 wrapped 标志区分——软换行只把两行首尾相接（且不 trim 行尾，避免刚好
  在行尾的空格被吃掉导致粘连），真实换行才补 `\n`
- 在 iShell 终端里跑 Claude Code 等 TUI 时大量快捷键失灵：按键编码此前只处理了
  `Ctrl+字母`，其余修饰组合一律被丢弃或干脆不发。现补齐 xterm 标准编码——
  - **Shift+Tab** 发 `CSI Z`（back-tab），此前发的是普通 `\t`，于是 Claude Code 里用来切换
    权限模式（plan / auto-accept）的 Shift+Tab 变成了 Tab 补全
  - **Shift/Alt+Enter** 发 `ESC CR`，用作"换行但不提交"（裸 xterm 编码下它和回车都是 CR、
    无法区分，这也正是 Claude Code 的 `/terminal-setup` 要给别的终端写入的映射；iShell 自己
    就是终端，现已内建，用户无需再配）
  - **修饰键 + 方向键 / Home / End** 发 `CSI 1;<m> X`、**+ Insert/Delete/PageUp/PageDown**
    发 `CSI <n>;<m> ~`，此前修饰键被丢弃、降级成无修饰的普通方向键（Ctrl+←/→ 按词跳转、
    Shift+←/→ 选择因此失效）
  - **F1–F12**（含修饰）此前完全不发
  - **Alt+字母/数字** 发 `ESC <char>`（Meta 惯例：Alt+B/F 按词移动、Alt+D 删词），
    **Alt+Backspace** 发 `ESC DEL` 删词；macOS 上按住 Alt 时同时产生的文本事件（Option+B
    会输入 "∫"）随之抑制，即"Option 当 Meta 键"，但 AltGr（被报成 Ctrl+Alt）仍照常输入字符
  - **Ctrl+Space/[/\\/]//-** 发对应控制字符（NUL / ESC / FS / GS / US）

## [0.16.8] - 2026-07-16

### Fixed
- 编辑器保存在弱网/连接静默失联（如 Windows 上笔记本休眠唤醒、Wi-Fi 切换、VPN 抖动导致
  TCP 连接死掉但未收到 RST）时可能永久卡在"保存中"：此前保存请求发出后只等
  `FileSaved`/`FileSaveFailed`/`FileSaveConflict` 三种结果之一，没有任何超时兜底，
  远端 worker 迟迟不返回时标签会一直显示保存中、"珊瑚→绿"动画也卡在中途不再推进。
  现在保存关联一个 30 秒截止时间，超时则解锁标签为可重试状态并弹"保存超时，请检查网络
  连接"提示；每次保存分配一个独立 id，超时后姗姗来迟的旧结果会被安全识别并丢弃，
  不会误更新一个已经关闭或已重新保存过的标签
- 粘贴文本（尤其是从 Windows 记事本/Office/浏览器复制的内容，几乎总是 CRLF 换行）未做
  换行符归一化就直接插入编辑器内容，与内部"统一 LF"的约定不符，导致保存后的文件混入
  裸 `\r`、且可能让"内容是否变化"的判断被这些杂散字符干扰
- `copy_from_remote` 与 `copy_to_remote` 对"本地"的解析点不一致：前者一直是在 iShell GUI
  所在机器上落盘，后者是在运行 `ishell-mcp` 的调用方机器上——两个工具的"本地"其实不是
  同一台机器，导致"先 `copy_from_remote` 下载、再 `copy_to_remote` 读同一路径"这类中转
  操作会莫名报"文件不存在"。现在 `copy_from_remote` 也改为在调用方机器本地流式落盘，
  与 `copy_to_remote` 对称；目前流式模式仅支持单个文件，目录请多次调用或改用
  `run_command` 执行 `tar`/`rsync`
- AI/MCP 反向转发会在每次连接/重连时于远端主机 `$HOME` 根目录下新建一个
  `.ishell-mcp-<nonce>.sock` 文件，且从未清理，长期使用会在主目录堆积大量残留文件；
  现在统一放进 `~/.ishell-mcp/` 子目录，连接正常断开时主动删除本次注册的文件，
  另外每次连接前顺手清理这个子目录里 mtime 超过 24 小时的旧文件兜底崩溃场景
- AI/MCP 流式文件传输（`copy_to_remote`/`copy_from_remote`）此前直接以截断方式打开目标
  文件，一旦传输中途失败（断线/超时/源文件变化）就会在远端或调用方本机留下一个空/半截
  文件、破坏同名原文件；现在改为「写临时文件→校验字节数→原子换入」的事务写，失败即
  原文件分毫未动
- `copy_between_sessions` 直连模式下，多个拷贝并发写同一台目标主机的 `authorized_keys` 会
  互相踩：撤销临时信任的「读→改→写」若与另一次授信的追加/另一次撤销交错，会丢标记（临时
  公钥残留）或互相覆盖，且此前用的是固定临时文件名还会撞名；现在授信/撤销用同一把 `flock`
  串行化，临时文件名另加随机后缀双保险

### Added
- 新增 `copy_between_sessions` 工具：把一个已打开远端会话上的文件复制到另一个已打开远端
  会话，不需要先下载到本机再上传。优先尝试源主机直连目标主机（不经过运行 iShell 的机器
  中转，适合两台主机同一局域网/集群的场景）——会生成一个仅限本次使用的一次性密钥对，
  临时授权源主机免密连接目标主机，传输完成后立即撤销这个临时授权、删除临时密钥，不留
  长期可用的免密信任；如果直连不可行（网络不通、权限受限等），自动降级为经 iShell 进程
  内存中转（不落盘任何一方磁盘）。当前仅支持单个文件

### Changed
- AI/MCP 文件传输支持多文件同时传输：此前同一会话同一时刻只允许一个文件读写操作，多文件
  只能串行；现在放开为每会话最多约 16 个并发（SFTP 天然支持并发，worker 侧本就有并发+
  排队），AI 可以对同一会话并行发起多次 `copy_to_remote`/`copy_from_remote` 一次性传多个
  文件，超过上限才返回「并发已达上限」提示退避
- 文件列表工具栏调整为「后退 / 刷新 / 复制 / 收藏」四个按钮，收藏放在最后；后退图标从
  线箭头换成尖括号样式，收藏图标从系统字体的 ☆/★（笔画偏细、与其它按钮不搭）换成同一套
  phosphor 图标字体的 STAR（笔画与刷新/复制一致），已收藏用强调色标识

- `copy_to_remote`/`copy_from_remote` 的路径校验从只判断"是否绝对路径"加强为拒绝空文件名
  （如 `/`、`////`）和 `.`/`..` 路径段，避免早前相对路径被静默当成写到文件系统根目录
